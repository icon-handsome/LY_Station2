## 1. 背景与结论

### 1.1 现状

当前仓库是**第一工位（封头检测）**专用 IPC 程序，核心特征：

| 维度 | 第一工位现状 |
|------|-------------|
| 工件 | 仅封头 |
| 搬运 | 上下料机械臂 + 电磁吸盘（PLC 控，IPC 不算路径） |
| 视觉 | Mech-Eye 3D + CXP 双目 + 海康 C（未接入主扫描流） |
| 位姿 | LB（封头段）+ LBN（转盘段） |
| 检测 | 坡口 Po_Kou / 开孔 Hole / 厚度 Thickness |
| 扫描 | 12 段固定路径（转盘×2 + 封头 5 点×2） |
| 编排 | `StateMachine` 轮询 PLC `Trig_*` 触发器 |
| 配置 | `config.ini` + `scan_paths_config.json` |

架构上已有**部分可复用抽象**（触发器框架、多路径缓存、`inspectionType` 分支、HMI JSON 协议），但产品命名、寄存器语义、扫描几何、检测算法均绑定第一工位。

### 1.2 第二工位需求（摘要）

第二工位是**三种工件模式**共用一套 IPC/PLC/上位机框架：

| 模式 ID | 工件 | 搬运 | 第二工位特有设备 |
|---------|------|------|------------------|
| `MODE_END_CAP` | 封头 | 机械臂上下料 | 上/下料 3D 相机 |
| `MODE_CYLINDER_SEMI` | 圆筒半成品 | 吊装 | 伸缩杆、滚轮架、2×吊装碰撞雷达 |
| `MODE_SEMI_FINISHED` | 半成品/成品 | 吊装 | 1×吊装碰撞雷达（无伸缩杆） |

第二工位 IPC 还需承担（第一工位未实现或仅为占位）：

- 2D AI 有料判断（`Trig_StationMaterialCheck`）
- 上料/下料 3D 抓取点与下料点解算（`Trig_LoadGrasp` / `Trig_UnloadCalc`）
- 通孔位姿检测（`Trig_PoseCheck`，第一工位有框架但依赖 LB 点云）
- 吊装辅助定位（激光测距 + 2D 纵焊缝区域）
- 伸缩杆/吊装碰撞检测（激光雷达 → PLC 报警）
- AI 人员闯入监控（并行，非主流程步骤）
- 圆筒内部扫描编排（伸缩杆步进 + 滚轮旋转步进）

### 1.3 重构策略结论（推荐）

**推荐：单仓库 + 工位配置文件（Station Profile）**，而非 fork 两个独立项目。

理由：

1. `StateMachine` 触发器-应答模式与第二工位 PLC 交互模型一致。
2. `scan_paths_config.json` 已支持多路径，可扩展为「模式 × 路径」。
3. Modbus/HMI 框架可复用，仅需扩展寄存器语义与条件分支。
4. 第一工位已联调代码可作为 `MODE_END_CAP` 的回归基线。

不推荐直接 copy 仓库做第二工位：会造成双份 bug 修复、协议漂移、算法版本不一致。

## 3. 目标架构

### 3.1 分层结构（重构后）

```
┌─────────────────────────────────────────────────────────────┐
│  config: station_profile.ini + scan_paths_{mode}.json       │
└────────────────────────────┬────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  StationProfile（工位/模式策略）                              │
│  - 启用哪些 Trig_*  handler                                   │
│  - 启用哪些设备 Service                                       │
│  - 扫描段类型映射（turntable / endcap / telescopic / roller）│
└────────────────────────────┬────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  StateMachine（瘦身：调度 + 握手，业务下沉到 Handler）         │
└───┬──────────┬───────────┬────────────┬─────────────────────┘
    │          │           │            │
    ▼          ▼           ▼            ▼
 Vision      Tracking    Safety       AuxMotion
 Pipeline    Service     Monitor      (伸缩杆/吊装/碰撞)
```

### 3.2 新增核心抽象

#### 3.2.1 `StationProfile`（`modules/station/`）

```cpp
enum class StationId { FirstEndCap, SecondMultiMode };
enum class WorkModeId {
    ModeEndCap,           // MODE_END_CAP
    ModeCylinderSemi,   // MODE_CYLINDER_SEMI
    ModeSemiFinished,   // MODE_SEMI_FINISHED
};

struct StationProfile {
    StationId stationId;
    WorkModeId defaultMode;
    bool enableLoadGrasp;
    bool enableUnloadCalc;
    bool enablePoseCheck;
    bool enableTelescopicScan;
    bool enableHoistAssist;
    bool enableCollisionMonitor;
    int collisionRadarCount;          // 0 / 1 / 2
    QString scanPathsConfigFile;      // 按模式切换 JSON
    QVector<TriggerDefinition> activeTriggers;
};
```

由 `config.ini` 的 `[Station] stationId=2` 和 PLC `Station_WorkMode`(40003) / `ProductType`(40013) 共同决定运行时模式。

#### 3.2.2 `ITaskHandler`（从 `StateMachine` 拆出）

将 `state_machine.cpp`（约 4000 行）中的 `execute*Task()` 迁入独立 Handler：

| Handler | 源函数 | 模式可用性 |
|---------|--------|------------|
| `LoadGraspHandler` | `executeLoadGraspTask` | `MODE_END_CAP` |
| `StationMaterialCheckHandler` | `executeStationMaterialCheckTask` | 全部 |
| `PoseCheckHandler` | `executePoseCheckTask` | `MODE_END_CAP` |
| `ScanSegmentHandler` | `executeScanSegmentTask` | 全部（内部分支） |
| `InspectionHandler` | `executeInspectionTask` | 全部 |
| `UnloadCalcHandler` | `executeUnloadCalcTask` | `MODE_END_CAP` |
| `SelfCheckHandler` | `executeSelfCheckTask` | 全部 |
| `CodeReadHandler` | `executeCodeReadTask` | 全部 |

`StateMachine` 仅保留：轮询、触发器选择、超时、PLC 读写、Handler 分发。

#### 3.2.3 `SegmentKind` 扩展（`scan_path_types.h`）

```cpp
enum class SegmentKind {
    TurntableLbn,      // 现有：转盘 + 2D + LBN
    EndCapLb,          // 现有：封头 + 3D + LB
    ExternalScan,      // 新增：扫描机械臂外部点位
    TelescopicStep,    // 新增：伸缩杆步进（如每 20cm）
    RollerCircumferential, // 新增：滚轮旋转步进（如每 10°）
};
```

`scan_paths_config.json` 每个 point 增加 `"segmentKind"` 字段，替代仅用 `needRotation` 隐式推断。

### 阶段 3：补齐封头模式真实业务（第二工位 `MODE_END_CAP`，2–3 周）

**目标**：把第一工位遗留的**占位实现**改为第二工位封头流程文档要求的真实能力。

| 触发器 | 现状 | 目标实现 |
|--------|------|----------|
| `Trig_StationMaterialCheck` | 写死 OK | 扫描机械臂到点位 1 → 2D 采集 → AI 判断封头有无 → `Res_StationMaterialCheck` |
| `Trig_LoadGrasp` | 环境变量/默认位姿 | 上料 3D 相机拍照 → 工控机解算抓取点 → 写 40118-40128 |
| `Trig_PoseCheck` | LB 点云位姿 | 点位 2 通孔扫描解算 + 位姿正确性判断 |
| `Trig_UnloadCalc` | 环境变量/默认位姿 | 下料 3D 相机 → 满料判断 + 下料点坐标 |
| `Trig_CodeRead` | stub 返回 "RD" | 海康 C / OCR 读编号 → 写入结果寄存器 |
| `Trig_Inspection` | 强制 `Res=1`（现场调试） | 可配置；投产关闭 `forceInspectionOk` |

#### 3.1 建议新增模块

```
modules/vision/
  material_detection_adapter.h/.cpp    # 2D AI 有料
  load_grasp_3d_adapter.h/.cpp       # 上料 3D 解算
  unload_calc_3d_adapter.h/.cpp        # 下料 3D 解算
```

#### 3.2 与第一工位关系

- 第一工位若也需要这些能力，共用同一 Handler，由 `StationProfile` 切换算法参数。
- `MODE_END_CAP` 扫描段可**初期复用** `station1_default.json` 的 12 段路径，后续按第二工位机械布局单独标定。

#### 3.3 验收标准

- [ ] 封头模式完整跑通 PDF 第 1 页流程（除机械臂运动外由 PLC 验证）
- [ ] 有料/无料、通孔错误、下料满料等分支 PLC 信号正确

```cpp
if (profile.enableTelescopicScan) {

```

### 阶段 5：圆筒模式扫描与检测（4–6 周，依赖算法）

**目标**：支持 `MODE_CYLINDER_SEMI` 与 `MODE_SEMI_FINISHED`。

#### 5.1 扫描路径配置示例（`station2_cylinder_semi.json`）

```json
{
  "scanPaths": [{
    "pathId": 1,
    "pathName": "圆筒-外扫+内窥",
    "inspectionType": "weld_defect",
    "points": [
      { "pointIndex": 1, "segmentKind": "external_scan", "pointName": "外扫-1" },
      { "pointIndex": 2, "segmentKind": "telescopic_step", "stepMm": 200, "pointName": "内纵缝-1" },
      { "pointIndex": 3, "segmentKind": "roller_circumferential", "stepDeg": 10, "pointName": "内环缝-1" },
      { "pointIndex": 4, "segmentKind": "turntable_lbn", "needRotation": true, "pointName": "转台-1" }
    ]
  }]
}
```

#### 5.2 `VisionPipelineService` 分支

| `segmentKind` | 采集组合 | 位姿算法 |
|---------------|----------|----------|
| `external_scan` | Mech-Eye 3D 和/或线扫 | 待定 |
| `telescopic_step` | 伸缩杆末端扫描仪 + 相机 | 无位姿或固定标定 |
| `roller_circumferential` | 同上 + 滚轮角度 | 无位姿 |
| `turntable_lbn` | 复用现有 LBN 路径 | LBN |

#### 5.3 检测算法

| 模式 | 建议 |
|------|------|
| `MODE_CYLINDER_SEMI` | 新增 `InspectionType::WeldDefect` / `LongitudinalWeld` / `CircumferentialWeld`，对接第三方或自研 |
| `MODE_SEMI_FINISHED` | 以外部扫描 + 缺陷/编号为主，可弱化 LB/LBN |
| `MODE_END_CAP` | 保持 Po_Kou / Hole / Thickness |

`TrackingService` 改为策略工厂：`InspectionStrategy::create(inspectionType, workMode)`。

#### 5.4 验收标准

- [ ] `MODE_CYLINDER_SEMI` 完整扫描序列跑通（采集+缓存+握手）
- [ ] `MODE_SEMI_FINISHED` 无伸缩杆依赖
- [ ] 检测失败 → 「重新吊装测量」状态字通知 PLC

## 6. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| `state_machine.cpp` 重构引入回归 | 第一工位联调失败 | 阶段 2 先搬移不改逻辑；`stationId=1` CI 回归 |

| 单人维护 4000 行状态机 | 重构成本高 | 必须阶段 2 Handler 化，否则后续不可维护 |
| 配置膨胀 | 运维困难 | 每模式独立 profile，禁止在一个 JSON 混写三种模式 |

## 8. 第一阶段实施建议（可立即开工）

若资源有限，建议按以下最小顺序执行：

1. **本周**：完成阶段 0 的 `Station_WorkMode` 枚举对齐（与 PLC 开会 1 次）。
2. **第 1–2 周**：阶段 1 — 加 `[Station] stationId`，迁移 `scan_paths_config.json`，保证 `stationId=1` 零回归。
3. **第 3–4 周**：阶段 2 — 先拆 `SelfCheck` + `LoadGrasp` + `StationMaterialCheck` 三个 Handler。
4. **第 5 周起**：实装 `StationMaterialCheck` 2D AI（三种模式都依赖，优先级最高）。

**不建议第一步就改 `executeScanSegmentTask`**：该函数与 Mech-Eye/CXP/LBN 耦合最深，应在 Handler 框架稳定后再动。
