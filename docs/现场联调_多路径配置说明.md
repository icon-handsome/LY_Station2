## 二、当前工作模式：PLC 按段触发

### 2.1 核心机制

**重要**：当前系统采用 **PLC 按段触发模式**，而非 IPC 自动多路径循环。

```
工作流程：
┌─────────────────────────────────────────────────────────┐
│ PLC 控制流程（每个工件）                                │
├─────────────────────────────────────────────────────────┤
│ 1. 路径1：PLC 循环发送 Trig_ScanSegment，段号 1..6     │
│    → IPC 响应每段采集，缓存点云到内存                   │
│                                                         │
│ 2. 路径2：PLC 循环发送 Trig_ScanSegment，段号 1..6     │
│    → IPC 响应每段采集，缓存点云到内存                   │
│                                                         │
│ 3. 路径3：PLC 循环发送 Trig_ScanSegment，段号 1..6     │
│    → IPC 响应每段采集，缓存点云到内存                   │
│                                                         │
│ 4. 综合检测：PLC 发送 Trig_Inspection                   │
│    → IPC 从内存取点云，调用蓝友算法，返回结果           │
└─────────────────────────────────────────────────────────┘
```

### 2.2 关键点

1. **路径切换由 PLC 控制**：
   - IPC 不知道当前是哪条路径
   - IPC 只响应 PLC 的 `Trig_ScanSegment` 触发
   - 每次触发时，PLC 通过 `ScanSegmentIndex` 寄存器告知段号（1-6）

2. **段号重复问题**：
   - 3 条路径都使用段号 1-6
   - IPC 的内存缓存 `m_segmentCaptureResults` 是按段号索引的 `QMap<int, CaptureResult>`
   - **问题**：路径2的段号1会覆盖路径1的段号1！

3. **当前实现的局限**：
   - IPC 只能缓存 **最多 20 个不同段号** 的点云（`kMaxPointCloudCacheSize=20`）
   - 如果段号重复，后面的会覆盖前面的
   - 综合检测时，蓝友只取 `config.ini [Tracking]` 配置的 3 个固定段号

## 三、能否直接配合 PLC 跑？

### 3.1 答案：**不能直接跑，需要适配**

**原因**：当前 IPC 代码按 **单路径、段号唯一** 设计，不支持多路径的段号重复。

### 3.2 问题分析

#### 问题1：段号冲突
```cpp
// state_machine.cpp 中的缓存结构
QMap<int, CaptureResult> m_segmentCaptureResults;  // key=段号

// 路径1：段号1-6 → 缓存 [1,2,3,4,5,6]
// 路径2：段号1-6 → 覆盖 [1,2,3,4,5,6]  ❌ 路径1的数据丢失！
// 路径3：段号1-6 → 覆盖 [1,2,3,4,5,6]  ❌ 路径2的数据丢失！
```

#### 问题2：综合检测取点逻辑
```cpp
// tracking_service.cpp
// 蓝友固定取 config.ini [Tracking] 的 3 个段号
firstStationOuterSegmentIndex=1   // 外圈
firstStationInnerSegmentIndex=2   // 内圈
firstStationHoleSegmentIndex=3    // 孔

// 多路径时，段号1可能来自路径1、路径2或路径3，无法区分！
```

## 四、需要的适配方案

### 方案A：修改段号编码（推荐，改动最小）

#### 4.1 原理
将 **路径ID** 编码到段号中，避免冲突：
```
段号编码 = 路径ID * 100 + 点位索引

路径1：段号 101, 102, 103, 104, 105, 106
路径2：段号 201, 202, 203, 204, 205, 206
路径3：段号 301, 302, 303, 304, 305, 306
```

#### 4.2 需要修改的地方

**1. PLC 侧（你需要改）**：
```
路径1循环：ScanSegmentIndex = 101..106
路径2循环：ScanSegmentIndex = 201..206
路径3循环：ScanSegmentIndex = 301..306
```

**2. IPC 侧（我帮你改）**：
- `config.ini [Tracking]` 段号配置：
  ```ini
  firstStationOuterSegmentIndex=101  # 路径1点1
  firstStationInnerSegmentIndex=202  # 路径2点2
  firstStationHoleSegmentIndex=303   # 路径3点3
  ```

- `scan_paths_config.json` 点位索引：
  ```json
  "pathId": 1,
  "points": [
    { "pointIndex": 101, ... },
    { "pointIndex": 102, ... },
    ...
  ]
  ```

- 代码验证逻辑：允许段号 > 100

#### 4.3 优点
- 改动最小，不需要重构缓存结构
- PLC 和 IPC 都能清楚知道当前是哪条路径的哪个点
- 兼容现有的单路径逻辑

---

### 方案B：增加路径上下文（复杂，不推荐现场联调时用）

#### 4.1 原理
在 Modbus 协议中增加 **当前路径ID** 寄存器，IPC 缓存改为二维索引。

#### 4.2 需要修改的地方
- Modbus 协议：新增 `CurrentPathId` 寄存器
- IPC 缓存结构：`QMap<int, QMap<int, CaptureResult>>`（路径ID → 段号 → 点云）
- 综合检测逻辑：支持多路径融合

#### 4.3 缺点
- 改动大，需要修改协议、代码、测试
- 现场联调时间紧，风险高

## 八、常见问题

### Q1: 为什么不能直接用段号 1-6？
**A**: 因为 IPC 的缓存是按段号索引的，3 条路径都用 1-6 会互相覆盖。

### Q2: 段号编码会影响 PLC 程序复杂度吗？
**A**: 影响很小，只需在循环中加上路径偏移：
```
FOR i := 1 TO 6 DO
    ScanSegmentIndex := PathID * 100 + i;  // PathID=1,2,3
    Trig_ScanSegment := 1;
    // 等待 IPC 响应...
END_FOR
```

### Q3: 如果只跑单路径，需要改段号吗？
**A**: 不需要。如果只跑路径1，保持段号 1-6 即可。

### Q4: 综合检测时，蓝友能融合 3 条路径的所有点云吗？
**A**: 当前蓝友只支持 **3 段点云**（外/内/孔），不支持 18 段（3路径×6点）融合。
需要你确认：
- 是否每条路径单独检测？
- 还是 3 条路径融合后统一检测？
